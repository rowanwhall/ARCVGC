# Settings Architecture

## Shared layer
- **`SettingsRepository`** (`shared/.../data/`): Manages all app settings. Constructor: `SettingsRepository(storage: SettingsStorage, cacheStorage: CatalogCacheStorage? = null, favoritesRepository: FavoritesRepository? = null, appConfigRepository: AppConfigRepository? = null)`. Each setting has its own `MutableStateFlow` (e.g., `_showWinnerHighlight`, `_selectedThemeId`, `_darkModeId`, `_preferredFormatId`) plus a combined `settingItems: StateFlow<List<SettingItem>>` that auto-updates when any setting (or the appConfig default format) changes. Methods: `setBooleanSetting(key, value)` / `setIntSetting(key, value)` (generic dispatchers), `performAction(key)` (one-shot actions like cache clearing), `getSettingItems()` / `isShowWinnerHighlightEnabled()` / `getSelectedThemeId()` / `getDarkModeId()` / `getPreferredFormatId()` / `getEffectivePreferredFormatId()` (snapshots for iOS interop). `getEffectivePreferredFormatId()` returns the user's pick if set, otherwise the config's current default (or 1 if neither is available) — consumers that default to the config format should use this instead of `AppConfigRepository.getDefaultFormatId()`.
- **`SettingsStorage`** (`expect`/`actual`): `getBoolean`/`putBoolean`/`getInt`/`putInt` — Android uses `SharedPreferences`, iOS uses `NSUserDefaults`, Web uses `localStorage` via `@JsFun`
- **`SettingItem`** (`shared/.../ui/model/`): Sealed class with six subclasses: `DarkModeChoice` (key, title, subtitle, selectedModeId), `Toggle` (key, title, subtitle, isEnabled), `ColorChoice` (key, title, subtitle, selectedThemeId), `FormatChoice` (key, title, subtitle, selectedFormatId, defaultFormatId — `selectedFormatId == 0` means "follow config default"), `Action` (key, title, subtitle, confirmationMessage — triggers one-shot action), `Link` (key, title, subtitle, url — opens external URL in browser). Metadata defined once in shared `buildSettingItems()` — platforms never hardcode setting titles/subtitles
- **`DarkModeOption`** (`shared/.../ui/model/`): Enum with `System` (0), `Light` (1), `Dark` (2). `fromId()` companion for safe lookup.
- **Adding a new boolean setting**: Add a `MutableStateFlow` + setter in `SettingsRepository`, add to `buildSettingItems()`, add key routing in `setBooleanSetting()` — no platform UI changes needed. For new `SettingItem` subclass types, each platform's settings UI needs a new rendering branch.

## Android
- `SettingsRepository` interface + `SettingsRepositoryImpl` (Hilt-injectable, delegates to shared). Passes `CatalogCacheStorage(context)` + `FavoritesRepository` + `AppConfigRepository` to shared repo. Exposes `settingItems: StateFlow<List<SettingItem>>`, `showWinnerHighlight: StateFlow<Boolean>`, `selectedThemeId: StateFlow<Int>`, `darkModeId: StateFlow<Int>`, `preferredFormatId: StateFlow<Int>`, `performAction(key)`. Also exposes `val shared: SharedSettingsRepository` so that shared-module consumers (`SearchViewModel`, `ContentListViewModel`) can hand the shared instance to `SearchLogic` / `ContentListLogic` directly — no unsafe downcast, and test fakes can supply a real `SharedSettingsRepository(FakeSettingsStorage())`.
- `SettingsViewModel` (`@HiltViewModel`): Injects `SettingsRepository` + all 4 catalog repos + `AppConfigRepository`. `performAction(key)` calls `settingsRepository.performAction()` then `reload()` on all 4 catalog repos
- `SettingsPage`: Collects `settingItems` flow, renders each via `when`: `DarkModeChoiceSettingRow` (tappable -> `DarkModePickerSheet`), `ToggleSettingRow` (switch), `ColorChoiceSettingRow` (tappable with color swatch -> `ThemePickerSheet`), `FormatChoiceSettingRow` (tappable -> `PreferredFormatPickerSheet`; shows `CircularProgressIndicator` and is disabled while the format catalog is loading), `ActionSettingRow` (tappable -> confirmation `AlertDialog`), `LinkSettingRow` (tappable -> opens URL via `Intent(ACTION_VIEW)`)

## iOS
- `SettingsStore` (`@MainActor ObservableObject`): Wraps shared `SettingsRepository` (passes `CatalogCacheStorage()` + `FavoritesRepository` + `AppConfigRepository` to constructor), exposes `@Published settingItems`, `@Published showWinnerHighlight`, `@Published selectedThemeId`, `@Published darkModeId`, `@Published preferredFormatId`, `@Published themeColor`, computed `colorSchemeOverride: ColorScheme?` + `effectivePreferredFormatId: Int32` (snapshot). Methods: `setBooleanSetting()`, `setIntSetting()`, `performAction()`. `syncState()` refreshes all after mutations; an `AsyncSequence` task observes `repo.settingItems` so app-config-driven defaults (e.g., `SettingItem.FormatChoice.defaultFormatId`) refresh once the config loads.
- `SettingsView`: Takes `settingsStore` + optional `catalogStore`. Iterates `settingsStore.settingItems` with SKIE `switch onEnum(of:)` — `.darkModeChoice` renders tappable row -> `DarkModePickerSheet`, `.toggle` renders `Toggle`, `.colorChoice` renders tappable row -> `ThemePickerSheet`, `.formatChoice` renders `FormatChoiceSettingRow` -> `PreferredFormatPickerSheet` (uses `catalogStore.formatItems` + `formatLoading`; shows a `ProgressView` and is disabled while loading), `.action` renders tappable row -> confirmation `.alert()` that calls `performAction()` + `catalogStore?.reload()`, `.link` renders SwiftUI `Link` that opens URL in browser

## Web
- Uses shared `SettingsRepository` directly via `DependencyContainer` (passes `cacheStorage`, `favoritesRepository`, `appConfigRepository` to constructor)
- `SettingsPage`: Collects `settingItems` flow + `formatCatalogRepository.state`, same `when` pattern as Android — `DarkModeChoiceSettingRow` (-> `DarkModePickerDialog`), `ToggleSettingRow`, `ColorChoiceSettingRow` (-> `ThemePickerDialog`), `FormatChoiceSettingRow` (-> `PreferredFormatPickerDialog`; shows `CircularProgressIndicator` and is disabled while the format catalog is loading), `ActionSettingRow` (-> confirmation `AlertDialog` that calls `performAction()` + `reload()` on all 4 catalog repos from `DependencyContainer`), `LinkSettingRow` (opens URL via `window.open()`)

## Preferred format resolution
- `SettingItem.FormatChoice.selectedFormatId == SettingsRepository.USE_DEFAULT_FORMAT` (value `0`) is the sentinel for "follow config default". The picker surfaces a `VGC Default - X` option that writes that sentinel, plus every format from the catalog (so the default is intentionally listed twice — once dynamically via `VGC Default`, once as a pinned choice).
- `SettingItem.FormatChoice.defaultFormatId == 0` means the app config hasn't resolved yet (the value is read via `appConfigRepository?.getConfig()?.defaultFormat?.id ?: 0`, *not* `getDefaultFormatId()`, which would collapse `null` config to `1`). The UI key the row off `defaultFormatId == 0 || catalogLoading` to disable interaction and show a spinner.
- `ContentListLogic` and `SearchLogic` both take an optional `SettingsRepository`; when present, any path that previously fell back to `AppConfigRepository.getDefaultFormatId()` (Home, Top Pokémon, un-threaded Pokemon/Player drill-downs, Search's initial default-format selection) now uses `getEffectivePreferredFormatId()` instead. Callers who thread an explicit `formatId` through navigation are unaffected — only defaults are overridden. `SearchLogic` additionally accepts a `StateFlow<List<FormatUiModel>>` so it can look up the display name for the preferred format's ID; if the catalog isn't provided, it falls back to the old config-default behaviour.
- **Live changes are intentional no-ops.** Once a `ContentListLogic` / `SearchLogic` instance has resolved its default, later changes to the preferred format do not retroactively update already-open screens — the user sees the new preference the next time they navigate back to Home / Search / Top Pokémon. This matches how we treat other config-driven defaults and avoids yanking content out from under users mid-scroll.
- **iOS SearchLogic wiring.** iOS's `CatalogStore` uses Swift `@Published` rather than Kotlin `StateFlow`, so `SearchViewModel` on iOS passes `settingsRepository: nil, formatCatalogFlow: nil` and applies the preferred format from the Swift side instead: `SearchView` calls `viewModel.applyPreferredFormatIfPossible()` from `.task` and on `.onChange(of: catalogStore.formatItems)` / `settingsStore.preferredFormatId`, which resolves the id and calls `SearchLogic.setDefaultFormat()` (a no-op if the user has already picked a format).
- **Format dropdown pinning.** `FormatSorter.sorted` is now called with the user's effective preferred format id (falling back to the config default), so the user's chosen format appears first in every format dropdown (Home, Search, Top Pokémon, Pokemon/Player drill-downs).
