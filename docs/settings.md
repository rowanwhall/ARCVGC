# Settings Architecture

## Shared layer
- **`SettingsRepository`** (`shared/.../data/`): Manages all app settings. Constructor: `SettingsRepository(storage: SettingsStorage, cacheStorage: CatalogCacheStorage? = null, favoritesRepository: FavoritesRepository? = null, appConfigRepository: AppConfigRepository? = null)`. Each setting has its own `MutableStateFlow` (e.g., `_showWinnerHighlight`, `_selectedThemeId`, `_darkModeId`, `_preferredFormatId`) plus a combined `settingSections: StateFlow<List<SettingsSection>>` that auto-updates when any setting (or the appConfig default format) changes. Methods: `setBooleanSetting(key, value)` / `setIntSetting(key, value)` (generic dispatchers), `performAction(key)` (one-shot actions like cache clearing), `getSettingSections()` / `isShowWinnerHighlightEnabled()` / `getSelectedThemeId()` / `getDarkModeId()` / `getPreferredFormatId()` / `getEffectivePreferredFormatId()` (snapshots for iOS interop). `getEffectivePreferredFormatId()` returns the user's pick if set, otherwise the config's current default (or 1 if neither is available) — consumers that default to the config format should use this instead of `AppConfigRepository.getDefaultFormatId()`.
- **`SettingsStorage`** (`expect`/`actual`): `getBoolean`/`putBoolean`/`getInt`/`putInt` — Android uses `SharedPreferences`, iOS uses `NSUserDefaults`, Web uses `localStorage` via `@JsFun`
- **`SettingItem`** (`shared/.../ui/model/`): Sealed class with six subclasses: `DarkModeChoice` (key, title, subtitle, selectedModeId), `Toggle` (key, title, subtitle, isEnabled), `ColorChoice` (key, title, subtitle, selectedThemeId), `FormatChoice` (key, title, subtitle, selectedFormatId, defaultFormatId — `selectedFormatId == 0` means "follow config default"), `Action` (key, title, subtitle, confirmationMessage — triggers one-shot action), `Link` (key, title, subtitle, url — opens external URL in browser). Metadata defined once in shared `buildSettingSections()` — platforms never hardcode setting titles/subtitles
- **`SettingsSection`** (`shared/.../ui/model/`): `data class SettingsSection(title: String, items: List<SettingItem>)`. Groups related settings into named sections: `Appearance` (Dark Mode, Theme Color), `Behavior` (Preferred Format, Highlight Winner), `Data` (Clear Favorites, Invalidate Cache), `Links` (Privacy Policy, Terms of Service). To add a new section, return another entry from `buildSettingSections()`.
- **`DarkModeOption`** (`shared/.../ui/model/`): Enum with `System` (0), `Light` (1), `Dark` (2). `fromId()` companion for safe lookup.
- **Adding a new boolean setting**: Add a `MutableStateFlow` + setter in `SettingsRepository`, add to the appropriate section inside `buildSettingSections()`, add key routing in `setBooleanSetting()` — no platform UI changes needed. For new `SettingItem` subclass types, each platform's settings UI needs a new rendering branch.

## Android
- `SettingsRepository` interface + `SettingsRepositoryImpl` (Hilt-injectable, delegates to shared). Passes `CatalogCacheStorage(context)` + `FavoritesRepository` + `AppConfigRepository` to shared repo. Exposes `settingSections: StateFlow<List<SettingsSection>>`, `showWinnerHighlight: StateFlow<Boolean>`, `selectedThemeId: StateFlow<Int>`, `darkModeId: StateFlow<Int>`, `preferredFormatId: StateFlow<Int>`, `performAction(key)`. Also exposes `val shared: SharedSettingsRepository` so that shared-module consumers (`SearchViewModel`, `ContentListViewModel`) can hand the shared instance to `SearchLogic` / `ContentListLogic` directly — no unsafe downcast, and test fakes can supply a real `SharedSettingsRepository(FakeSettingsStorage())`.
- `SettingsViewModel` (`@HiltViewModel`): Injects `SettingsRepository` + all 4 catalog repos + `AppConfigRepository`. `performAction(key)` calls `settingsRepository.performAction()` then `reload()` on all 4 catalog repos
- `SettingsPage`: Collects `settingSections` flow; iterates each `SettingsSection`, emitting a `SettingsSectionHeader` (uppercased `onSurfaceVariant` label) followed by a `SettingsSectionCard` (rounded `Surface` with a 1dp `outlineVariant` border; both page and card use `surface` so the card's outline is what visually distinguishes it). Inside each card, rows are separated by `HorizontalDivider(outlineVariant)` and rendered via `when`: `DarkModeChoiceSettingRow` (tappable -> `DarkModePickerSheet`), `ToggleSettingRow` (switch), `ColorChoiceSettingRow` (tappable with color swatch -> `ThemePickerSheet`), `FormatChoiceSettingRow` (tappable -> `PreferredFormatPickerSheet`; shows `CircularProgressIndicator` and is disabled while the format catalog is loading), `ActionSettingRow` (tappable -> confirmation `AlertDialog`), `LinkSettingRow` (tappable -> opens URL via `Intent(ACTION_VIEW)`)

## iOS
- `SettingsStore` (`@MainActor ObservableObject`): Wraps shared `SettingsRepository` (passes `CatalogCacheStorage()` + `FavoritesRepository` + `AppConfigRepository` to constructor), exposes `@Published settingSections`, `@Published showWinnerHighlight`, `@Published selectedThemeId`, `@Published darkModeId`, `@Published preferredFormatId`, `@Published themeColor`, computed `colorSchemeOverride: ColorScheme?` + `effectivePreferredFormatId: Int32` (snapshot). Methods: `setBooleanSetting()`, `setIntSetting()`, `performAction()`. `syncState()` refreshes all after mutations; an `AsyncSequence` task observes `repo.settingSections` so app-config-driven defaults (e.g., `SettingItem.FormatChoice.defaultFormatId`) refresh once the config loads.
- `SettingsView`: Takes `settingsStore`, `catalogStore`, and `favoritesStore` — all `@ObservedObject` so SwiftUI re-renders the row when `catalogStore.formatLoading` flips back to `false` after `reload()` (e.g., post-Invalidate Cache). Wraps content in a `ScrollView { VStack }` with `systemBackground` page bg. Each `SettingsSection` emits a `sectionHeader` (uppercased secondary label) followed by a `sectionCard` — a `VStack` with `systemBackground` background, rounded corners (`AppTokens.cardCornerRadius`), and a 1pt `Color(.opaqueSeparator)` overlay border (same as `BattleCardView`). Both page and card share `systemBackground` so the border is what visually distinguishes the card. Inside each card, rows are separated by an `opaqueSeparator`-colored 1pt rectangle (matches the card border) and rendered via SKIE `switch onEnum(of:)` — `.toggle` renders title/subtitle + native `Toggle`; `.darkModeChoice`/`.colorChoice`/`.action` render via the shared `tappableRow` helper (uses `.contentShape(Rectangle()).onTapGesture` plus `.accessibilityAddTraits(.isButton)` so VoiceOver still announces them as buttons after the move off `List`); `.link` is a `SwiftUI.Link` wrapping the same row layout (preserves long-press preview/share menu); `.formatChoice` renders `FormatChoiceSettingRow` (shows `ProgressView` while catalog is loading). Action confirmation is via `.alert()` (calls `performAction()` + `catalogStore?.reload()` + `favoritesStore.syncState()`).

## Web
- Uses shared `SettingsRepository` directly via `DependencyContainer` (passes `cacheStorage`, `favoritesRepository`, `appConfigRepository` to constructor)
- `SettingsPage`: Collects `settingSections` flow + `formatCatalogRepository.state`, same sectioned-card pattern as Android — iterates sections, emitting a `SettingsSectionHeader` and a `SettingsSectionCard` per section (page bg + card bg both use `surface`; the card's `outlineVariant` border distinguishes it). Rows within each card are separated by `HorizontalDivider` and rendered via `when` matching each `SettingItem` type: `DarkModeChoiceSettingRow` (-> `DarkModePickerDialog`), `ToggleSettingRow`, `ColorChoiceSettingRow` (-> `ThemePickerDialog`), `FormatChoiceSettingRow` (-> `PreferredFormatPickerDialog`; shows `CircularProgressIndicator` and is disabled while the format catalog is loading), `ActionSettingRow` (-> confirmation `AlertDialog` that calls `performAction()` + `reload()` on all 4 catalog repos from `DependencyContainer`), `LinkSettingRow` (opens URL via `window.open()`)

## Preferred format resolution
- `SettingItem.FormatChoice.selectedFormatId == SettingsRepository.USE_DEFAULT_FORMAT` (value `0`) is the sentinel for "follow config default". The picker surfaces a `VGC Default - X` option that writes that sentinel, plus every format from the catalog (so the default is intentionally listed twice — once dynamically via `VGC Default`, once as a pinned choice).
- `SettingItem.FormatChoice.defaultFormatId == 0` means the app config hasn't resolved yet (the value is read via `appConfigRepository?.getConfig()?.defaultFormat?.id ?: 0`, *not* `getDefaultFormatId()`, which would collapse `null` config to `1`). The UI key the row off `defaultFormatId == 0 || catalogLoading` to disable interaction and show a spinner.
- `ContentListLogic` and `SearchLogic` both take an optional `SettingsRepository`; when present, any path that previously fell back to `AppConfigRepository.getDefaultFormatId()` (Home, Top Pokémon, un-threaded Pokemon/Player drill-downs, Search's initial default-format selection) now uses `getEffectivePreferredFormatId()` instead. Callers who thread an explicit `formatId` through navigation are unaffected — only defaults are overridden. `SearchLogic` additionally accepts a `StateFlow<List<FormatUiModel>>` so it can look up the display name for the preferred format's ID; if the catalog isn't provided, it falls back to the old config-default behaviour.
- **Live changes are intentional no-ops.** Once a `ContentListLogic` / `SearchLogic` instance has resolved its default, later changes to the preferred format do not retroactively update already-open screens — the user sees the new preference the next time they navigate back to Home / Search / Top Pokémon. This matches how we treat other config-driven defaults and avoids yanking content out from under users mid-scroll.
- **iOS SearchLogic wiring.** iOS's `CatalogStore` uses Swift `@Published` rather than Kotlin `StateFlow`, so `SearchViewModel` on iOS passes `settingsRepository: nil, formatCatalogFlow: nil` and applies the preferred format from the Swift side instead: `SearchView` calls `viewModel.applyPreferredFormatIfPossible()` from `.task` and on `.onChange(of: catalogStore.formatItems)` / `settingsStore.preferredFormatId`, which resolves the id and calls `SearchLogic.setDefaultFormat()` (a no-op if the user has already picked a format).
- **Format dropdown pinning.** `FormatSorter.sorted` is now called with the user's effective preferred format id (falling back to the config default), so the user's chosen format appears first in every format dropdown (Home, Search, Top Pokémon, Pokemon/Player drill-downs) and in the Settings preferred-format picker itself. After the pinned format, remaining formats are partitioned by `isOfficial`: those matching the pinned format's `isOfficial` value come first, then the rest, each group sorted descending by ID.
